  <name>Task 1: Description re-optimization endpoints (GET + POST + push)</name>
  <files>src/app/api/reoptimize/description/route.ts, src/app/api/reoptimize/description/push/route.ts</files>
  <action>
Create two route files for description re-optimization. These follow the exact same pattern as the onboarding description endpoints but are separate routes so they work independently of the wizard.

**GET /api/reoptimize/description?profileId=X** -- Returns:
- `currentGBPDescription`: fetched live from GBP via `fetchCurrentDescription`
- `savedDescription`: latest ProfileDescription record from DB (if any) with id, content, isApproved, isPushed, pushedAt
- `keywords`: array of keyword strings from ProfileKeyword for this profile
- Auth check via getServerSession, 401 if not authenticated
- 400 if profileId missing, 404 if profile not found

**POST /api/reoptimize/description** -- Generates new AI description:
- Accepts `{ profileId }` in body
- Loads profile (name, category, address), keywords from ProfileKeyword, cities from ProfileCity
- Calls `generateDescription()` from `@/lib/description-generator`
- Returns `{ description: string }`
- Same auth/validation pattern as GET

**POST /api/reoptimize/description/push** -- Pushes approved description to GBP:
- Accepts `{ profileId, content }` in body
- Validates content is non-empty string, max 750 chars
- Upserts ProfileDescription record (find existing, update or create) with isApproved: true
- Calls `pushDescriptionToGBP` from `@/lib/google-business-info`
- On success: updates record with isPushed: true, pushedAt: new Date()
- Returns `{ success: boolean, error?: string, description?: record }`
- This is essentially the same logic as `/api/onboarding/description/push/route.ts` -- replicate the pattern exactly.

All three handlers follow the established auth pattern: `getServerSession(authOptions)`, return 401 if no session. Profile lookup includes googleAccountId and locationName for GBP calls.

<task type="auto">
  <name>Task 2: Services re-optimization endpoints (GET + POST + push)</name>
  <files>src/app/api/reoptimize/services/route.ts, src/app/api/reoptimize/services/push/route.ts</files>
  <action>
Create two route files for services re-optimization. These follow the same pattern as the onboarding services endpoints.

**GET /api/reoptimize/services?profileId=X** -- Returns:
- `currentGBPServices`: array of structured service info fetched live from GBP via `fetchStructuredServices` (serviceTypeId, displayName)
- `savedServices`: array of ProfileService records from DB for this profile (id, serviceName, description, isStructured, isApproved, isPushed, pushedAt)
- `availableServices`: same as currentGBPServices (the available structured services for the category) -- this gives the frontend the list of services to offer for re-selection
- Auth check, 401/400/404 as above

**POST /api/reoptimize/services** -- Generates new AI service descriptions:
- Accepts `{ profileId, serviceNames: string[] }` in body
- Validates serviceNames is non-empty array, max 20 items
- Loads profile, keywords from ProfileKeyword, cities from ProfileCity
- Calls `generateServiceDescriptions()` from `@/lib/service-generator`
- Returns `{ services: Array<{ serviceName, description }> }`
- Same pattern as `/api/onboarding/services/generate/route.ts`

**POST /api/reoptimize/services/push** -- Pushes approved services to GBP:
- Accepts `{ profileId }` in body
- Loads approved ProfileService records from DB
- Returns 400 if no approved services
- Fetches current GBP services via `fetchCurrentServices` for merge
- Logs pre-push snapshot (same pattern as onboarding push)
- Merges: for structured services, find by serviceTypeId and replace; for custom services, append as freeFormServiceItem
- Pushes merged list via `pushServicesToGBP`
- On success: updates all approved ProfileService records with isPushed: true, pushedAt: new Date()
- Returns `{ success: boolean, pushedCount?: number, error?: string }`
- This is essentially the same logic as `/api/onboarding/services/push/route.ts` -- replicate the merge logic exactly.

Additionally, the POST /api/reoptimize/services route should also save the generated services to the DB (upsert ProfileService records by profileId+serviceName), so they persist for later push. Use `prisma.profileService.deleteMany({ where: { profileId } })` then `prisma.profileService.createMany()` to replace the full set, matching the onboarding pattern.
