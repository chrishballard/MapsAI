<task type="auto">
  <name>Task 1: Create ReoptimizeSection client component</name>
  <files>src/app/dashboard/profiles/[id]/reoptimize-section.tsx</files>
  <action>
Create a "use client" component `ReoptimizeSection` that provides side-by-side re-optimization for description and services.

**Props:** `{ profileId: string }`

**Layout:** Two collapsible sections stacked vertically, each in a white card with border:

**Section 1: Description Re-optimization**
- Header: "Description" with a "Re-optimize" button (uses Sparkles icon)
- Initially shows: current live GBP description on the left, last saved/pushed description on the right (or "No saved description" placeholder)
- When user clicks "Re-optimize":
  1. Calls POST /api/reoptimize/description with profileId
  2. Shows loading skeleton during generation
  3. On success: displays side-by-side comparison:
     - LEFT panel: "Current (Live on Google)" -- shows currentGBPDescription in a gray-50 bg readonly block
     - RIGHT panel: "AI Suggestion" -- shows generated description in an editable textarea
  4. Below the textarea: character counter (X/750), keyword coverage badges (same pattern as description-step.tsx)
  5. Action buttons: "Approve & Push to Google" (green-600), "Regenerate" (border button with RotateCcw icon)
- On push: calls POST /api/reoptimize/description/push, shows success banner, updates state
- Error handling: red banner with error message and retry button (same pattern as description-step.tsx)

**Section 2: Services Re-optimization**
- Header: "Services" with a "Re-optimize" button
- Initially shows: list of currently saved services with their descriptions and push status
- When user clicks "Re-optimize":
  1. First shows a service selection UI: checkboxes for available structured services (from GET endpoint), plus custom service input (same pattern as services-step.tsx selection phase)
  2. On "Generate Descriptions": calls POST /api/reoptimize/services with selected serviceNames
  3. Shows loading skeleton during generation
  4. On success: shows service cards with editable descriptions, approve/unapprove per card, "Approve All" button
  5. "Push All to Google" button: calls POST /api/reoptimize/services/push
- Each service card shows: service name, structured/custom badge, editable textarea, approve button, character count
- Same card design as services-step.tsx but without the wizard context

**Data loading:** On mount, call both GET endpoints in parallel to load current state. Show a loading spinner until both resolve.

**Styling:** Follow existing Tailwind patterns from the onboarding steps. Use the same icon imports (Sparkles, Loader2, CheckCircle2, AlertCircle, RotateCcw, Plus, X, Wrench) from lucide-react.

**Key difference from onboarding steps:** This component does NOT auto-generate on mount. User explicitly clicks "Re-optimize" to trigger generation. The initial state shows what is currently live/saved.
